## Боль заказчика

**~1000 сотрудников. 104 000 тикетов. Ноль структурированной базы знаний.**

Вся экспертиза IT-поддержки «Балтийского Берега» накоплена в MSSQL-системе IntraService — в виде сырых HTML-диалогов между сотрудниками и операторами. Никакой KB нет. Каждый новый инцидент — звонок или тикет к живому специалисту первой линии.

**Результат:** перегрузка 1-й линии, долгое ожидание, экспертиза недоступна рядовому сотруднику.

---

## Наше решение

Мы построили RAG-систему поверх существующей БД **без её изменения** (read-only). Система извлекает знания из трёх источников, строит векторный индекс и отвечает на новые вопросы — опираясь на то, как аналогичные проблемы решались раньше.

Пользователь пишет в Telegram. Бот отвечает за 2–5 секунд. Если уверенности нет — честно эскалирует к оператору.

---

## Как это работает

```
Сотрудник → Telegram
                │
                ▼
        [Очистка запроса]
        LLM вычленяет суть из эмоционального текста
                │
                ▼
        [Семантический поиск]
        93 000+ чанков: KB-статьи + тикеты + решения инженеров
                │
                ▼
        [YandexGPT]
        Формирует ответ по топ-6 найденным фрагментам
                │
                ▼
        [Проверка уверенности]
        score ≥ 0.45 → ответ + источник [KB / Тикет / Решение]
        score < 0.45 → честная эскалация к оператору
                │
                ▼
        [Admin-бот]
        Оператор видит карточку, ставит оценку 1–5★
                │
                ▼
        [Feedback loop]
        Оценка влияет на будущее ранжирование
```

## Ключевые фичи

### 1. Трёхисточниковый RAG-поиск

Один запрос проверяется сразу по трём коллекциям ChromaDB:

| Источник | Таблица MSSQL | Содержимое |
|---|---|---|
| KB-статьи | `KBDocument` | Официальные инструкции (~1 000 документов) |
| Тикеты | `Task.Comment` | Реальные Q&A-диалоги сотрудников и поддержки |
| Решения инженеров | `TaskExpenses.Comments` | Что именно сделал инженер при закрытии тикета |

Ответ находится даже если его нет в KB — он «живёт» в 104 000 диалогов.

### 2. Очистка запроса перед поиском

Перед семантическим поиском LLM вычленяет суть проблемы. Длинный, эмоциональный, с опечатками текст → короткий точный запрос → точнее поиск → лучше ответ.

### 3. Умная эскалация без галлюцинаций

Если топ-результат поиска имеет `score < 0.45` — система не придумывает ответ, а честно говорит «не знаю» и передаёт оператору. **Принцип: лучше не ответить, чем ответить неверно.**

### 4. Автоматическая классификация заявки

Без LLM-вызова, мгновенно — по ключевым словам:

- **7 категорий сервиса:** 1С и ERP, Сеть и VPN, IT-инфраструктура, Почта, Оргтехника, Доступ и права, Другое
- **4 приоритета:** Критичный, Высокий, Средний, Низкий

### 5. Source Attribution

Каждый ответ содержит ссылку на источник: `[KB]`, `[Тикет]` или `[Решение]`. Пользователь и оператор видят, откуда взят ответ.

### 6. Feedback Loop (кооперативное обучение)

Операторы ставят оценку 1–5★ каждому ответу прямо в Telegram-панели. Веса сохраняются в `data/weights.json` и влияют на ранжирование. Система становится лучше от работы команды — без переобучения модели.

### 7. Admin Telegram-бот

Отдельный бот для операторов первой линии:

- Уведомление о каждом запросе в реальном времени (карточка: вопрос, ответ, тип источника, флаг эскалации)
- Inline-кнопки: поставить оценку (1–5★), забрать заявку (`Claim`), ответить пользователю прямо из панели
- Команды: `/setweight`, `/resetweights`, `/addadmin`, `/removeadmin`

### 8. Веб-дашборд статистики

Только чтение — сайт предназначен для мониторинга, не для ввода вопросов:

- KPI: всего запросов, % автоматизации, % эскалаций, среднее время ответа
- Топ-10 категорий обращений из реальной БД
- Динамика объёма запросов по месяцам (последние 12 месяцев)
- Распределение оценок операторов (1–5)
- Таблица всех сессий с детальной фильтрацией

### 9. Страница анализа мышления нейросети

Каждый запрос имеет страницу `/analysis/{id}` — полная прозрачность:

1. Классификация запроса (сервис + приоритет)
2. Топ-3 найденных чанка с оценкой релевантности
3. Уровень уверенности в процентах
4. Текст итогового ответа
5. Решение: автоответ или эскалация — и почему именно так

## Конкурентные преимущества

**1. Нет KB — не проблема.**
Извлекаем знания из сырых Q&A-диалогов в 104к тикетах. Ни один конкурент не делает то же самое с такой же полнотой — большинство работают только с KB.

**2. Не галлюцинирует.**
Честная эскалация при score < 0.45. Система знает, когда не знает — и не врёт.

**3. Полная прозрачность решений.**
Страница анализа показывает весь ход «мышления» бота. Любое решение можно объяснить жюри, руководству или клиенту.

**4. Feedback loop без переобучения.**
Операторы ставят оценки → веса обновляются → ранжирование улучшается. Кооперативное обучение из коробки.

**5. Данные не покидают контур.**
ChromaDB и MSSQL — локально. YandexGPT API — единственная внешняя зависимость, легко заменяется на локальную LLM.

**6. Готово к расширению прямо сейчас.**
FastAPI `/ask` endpoint принимает `source: telegram | api`. Max, Битрикс, IntraService — подключаются без изменения логики.
